# project.launch 启动流程完整分析

## 启动流程图

```
1. roslaunch map_switch project.launch
   ├─ 启动 map_switch_node (地图切换节点)
   ├─ 启动 tf_publish_node (TF发布节点)
   └─ 启动 pointcloud_to_laserscan (点云转激光)

2. map_switch_node 初始化
   ├─ 读取 floors.yaml 配置（地图列表）
   ├─ 创建TCP服务器（监听6051端口）
   └─ 自动启动默认地图（map id=1）
       │
       └─> 调用 launchMap(1, 0, 0, 0, 0, -1)
           │
           ├─ 执行 start_map1.sh
           │   │
           │   └─> roslaunch publish_map1.launch
           │       ├─ 启动 fastlio_mapping_global (FAST_LIO定位)
           │       ├─ 启动 pcd_to_pointcloud (发布地图点云)
           │       ├─ 启动 map_server (发布2D地图)
           │       ├─ 启动 global_localization.py (全局定位)
           │       │   └─ 等待 /initialpose 话题 ⏳
           │       ├─ 启动 transform_fusion.py (TF融合)
           │       └─ 启动 tf_publish (TF发布)
           │
           ├─ 等待 /waiting_for_initial_pose 话题
           │   └─ global_localization.py 发布此话题
           │
           └─ 调用 PublishInitialPose(0, 0, 0)
               │
               └─> 执行命令: python3 publish_initial_pose.py 0 0 0 0 0 0
                   └─ 发布 /initialpose 话题 ✅
                       └─ global_localization.py 收到初始位姿
                           └─ 开始ICP匹配定位
                               └─ 发布 /map_to_odom_flag 话题
```

## 详细说明

### 阶段1: project.launch 启动
**文件**: `src/map_switch/launch/project.launch`

启动的节点：
- `map_switch_node` - 地图切换管理
- `tf_publish_node` - TF变换发布
- `pointcloud_to_laserscan` - 点云转激光扫描

**注意**: 此时**没有启动**任何定位相关的节点！

## 关键问题回答

### Q1: 启动 project.launch 后，会启动 publish_initial_pose.py 吗？

**答：会！但不是立即启动，而是按以下顺序：**

1. `project.launch` 启动 → `map_switch_node` 启动
2. `map_switch_node` 自动启动默认地图（map id=1）
3. 执行 `start_map1.sh` → 启动 `publish_map1.launch`
4. `publish_map1.launch` 启动 `global_localization.py`
5. `global_localization.py` 发布 `/waiting_for_initial_pose` 标志
6. `map_switch_node` 收到标志后调用 `publish_initial_pose.py` ✅

**时间线**:
```
0s:   project.launch 启动
1s:   map_switch_node 初始化完成
2s:   开始启动 start_map1.sh
3s:   publish_map1.launch 启动
5s:   global_localization.py 启动并发布等待标志
6s:   map_switch_node 调用 publish_initial_pose.py ✅
6-16s: publish_initial_pose.py 持续发布初始位姿
7s:   global_localization.py 收到初始位姿，开始定位
```

---

### Q2: global_localization_mid360.launch 在哪里用到？

**答：不会被 project.launch 使用！**

`global_localization_mid360.launch` 是一个**独立的测试launch文件**，用于：
- 直接测试FAST_LIO全局定位功能
- 不依赖地图切换系统
- 手动启动时使用

**使用场景**:
```bash
# 独立测试定位（不使用地图切换）
roslaunch fast_lio_global global_localization_mid360.launch
```

**与 project.launch 的区别**:

| 特性 | project.launch | global_localization_mid360.launch |
|------|----------------|-----------------------------------|
| 用途 | 生产环境，支持多地图切换 | 测试环境，单地图定位 |
| 地图切换 | ✅ 支持 | ❌ 不支持 |
| 初始位姿 | 通过 map_switch_node 调用 | 直接在launch中写死 |
| 启动方式 | 自动启动默认地图 | 手动启动 |

## 建议

### 当前问题：重启后坐标会变

**根本原因**: 
- 每次启动都使用固定的初始位姿 `(0, 0, 0)`
- ICP匹配可能收敛到不同的局部最优解

**解决方案**:

1. **修改 publish_map1.launch**（已完成）
   - 添加 `save_last_pose.py` - 自动保存位姿
   - 使用 `publish_saved_pose.py` - 恢复保存的位姿

2. **或者修改 map_switch_node.cpp**（刚才已修改）
   - 使用实时坐标而不是固定的 `(0,0,0)`
   - 但需要确保坐标转换正确

3. **编译并测试**
```bash
cd /home/orangepi/slam_ws
catkin_make
source devel/setup.bash
roslaunch map_switch project.launch
```
