# 多地图点管理修复总结

## 问题描述

用户反馈：
1. 打了地图切换点之后，在新地图上打点，之前在旧地图上打的点都没了
2. 新地图上打的点应该接在之前的点后面，seq连续
3. 只有按了"保存"按钮之后才应该往XML文件里写入，在此之前都应该放内存里

## 根本原因

**后端 `/api/switch_map` API 错误地清空了点**：
```cpp
// 错误的实现：
svr.Post("/api/switch_map", [](const httplib::Request& req, httplib::Response& res) {
    // ...
    save_xml_file();        // 保存当前地图的路径
    g_current_map_id++;
    g_points.clear();       // ❌ 错误：清空了所有点
    g_pixels.clear();       // ❌ 错误：清空了所有像素坐标
    // ...
});
```

这个API在"从SLAM获取"上传地图后被调用，导致内存中的点被清空。

## 修复方案

### 核心理念

**多地图点管理的正确逻辑**：
1. 所有地图的点都保存在**同一个内存数组** `g_points` 中
2. 每个点的 `mapnum` 字段标识它属于哪个地图
3. 地图切换点的 `mapnum` 存储**目标地图ID**
4. 只有用户点击"保存"按钮时，才将内存中的点写入XML文件
5. XML文件包含所有地图的点，通过 `mapnum` 字段区分

### 修复内容

#### 1. 修改 `/api/switch_map` API (`httpcopy/main.cpp`)

```cpp
// ---- 切换地图（仅用于上传地图后切换显示，不清空点）----
svr.Post("/api/switch_map", [](const httplib::Request& req, httplib::Response& res) {
    res.set_header("Access-Control-Allow-Origin", "*");
    std::string new_map_name = json_get(req.body, "map_name");
    if (new_map_name.empty()) new_map_name = "地图" + std::to_string(g_current_map_id + 1);
    
    // 🔴 注意：此API仅用于切换显示的地图，不清空点
    // 点应该一直保留在内存中，直到用户点击"保存"按钮
    // 如果需要清空点并开始新路径，应该使用单独的API
    
    std::cout << "切换显示地图: " << new_map_name << " (点不清空，保留在内存中)" << std::endl;
    
    std::ostringstream ss;
    ss << "{\"status\":\"ok\",\"map_id\":" << g_current_map_id 
       << ",\"map_name\":\"" << new_map_name << "\"}";
    res.set_content(ss.str(), "application/json");
});
```

**关键改动**：
- ❌ 删除了 `save_xml_file()` 调用
- ❌ 删除了 `g_points.clear()` 调用
- ❌ 删除了 `g_pixels.clear()` 调用
- ❌ 删除了 `g_current_map_id++` 调用
- ✅ 只是一个通知性的API，不修改任何状态

## 数据结构说明

### PathPoint 结构
```cpp
struct PathPoint {
    int seq;              // 点的序号（全局唯一，连续递增）
    double x;             // 世界坐标X
    double y;             // 世界坐标Y
    double yaw;           // 航向角
    int keypointflag;     // 点类型（0=普通点, 5=地图切换点, 等）
    int mapnum;           // 地图编号（普通点=当前地图ID，切换点=目标地图ID）
};
```

### 全局变量
```cpp
static std::vector<PathPoint> g_points;   // 所有地图的点都在这里
static std::vector<PixelPoint> g_pixels;  // 对应的像素坐标
static std::mutex g_points_mutex;         // 线程安全保护
static int g_current_map_id = 1;          // 当前显示的地图ID
```

## 完整工作流程

### 场景1: 单地图打点

```
1. 用户选择点类型（如"普通点"）
2. 点击"打点"按钮
   ↓
3. 前端调用 /api/get_position 获取当前位置
   ↓
4. 前端调用 /api/add_point 添加点
   - seq = g_points.size() + 1  (自动递增)
   - mapnum = pose.map_id (当前地图ID)
   ↓
5. 点添加到内存 g_points 中
   ↓
6. 用户继续打点...
   ↓
7. 用户点击"保存"按钮
   ↓
8. 前端调用 /api/save_points
   ↓
9. 后端调用 save_xml_file()
   - 将 g_points 序列化为XML
   - 写入文件 test1.xml
```

### 场景2: 多地图打点（含地图切换点）

```
地图1上打点：
1. 打点1: seq=1, mapnum=1, keypointflag=0
2. 打点2: seq=2, mapnum=1, keypointflag=0
3. 打地图切换点: seq=3, mapnum=2, keypointflag=5
   ↓
   触发SLAM地图切换（地图1 → 地图2）
   ↓
   等待SLAM重定位完成
   ↓
地图2上打点：
4. 打点4: seq=4, mapnum=2, keypointflag=0  ✅ seq连续
5. 打点5: seq=5, mapnum=2, keypointflag=0  ✅ seq连续
   ↓
用户点击"保存"：
   ↓
XML文件内容：
<Root>
  <path Seq="1">
    <mapnum>1</mapnum>  <!-- 地图1的点 -->
  </path>
  <path Seq="2">
    <mapnum>1</mapnum>  <!-- 地图1的点 -->
  </path>
  <path Seq="3">
    <mapnum>2</mapnum>  <!-- 地图切换点，目标地图2 -->
    <keypointflag>5</keypointflag>
  </path>
  <path Seq="4">
    <mapnum>2</mapnum>  <!-- 地图2的点 -->
  </path>
  <path Seq="5">
    <mapnum>2</mapnum>  <!-- 地图2的点 -->
  </path>
</Root>
```

## 关键点总结

### ✅ 正确的行为

1. **点的seq连续递增**：无论在哪个地图上打点，seq都是全局唯一且连续的
2. **点不会丢失**：切换地图后，旧地图的点仍然保留在内存中
3. **mapnum正确标识**：
   - 普通点：`mapnum = 当前地图ID`
   - 地图切换点：`mapnum = 目标地图ID`
4. **只有保存时写文件**：点击"保存"按钮才调用 `save_xml_file()`

### ❌ 错误的行为（已修复）

1. ~~切换地图时清空点~~ → 现在不清空
2. ~~切换地图时自动保存~~ → 现在不自动保存
3. ~~每个地图独立的点列表~~ → 现在所有点在同一个列表
